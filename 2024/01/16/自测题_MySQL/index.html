面试解答篇

  • 基础
    • 说一下MySQL的架构。
      • MySQL的架构包括连接层、服务层和存储层。连接层负责与客户端建立连接、授权认证、及连接管理。服务层包括SQL接口、查询解析、分析、优化、缓存以及内置函数等，负责SQL语句的解析、优化、以及执行计划的生成。存储层负责数据的存储和提取，其核心是存储引擎，MySQL支持多种存储引擎，每种引擎有其特性。
        • （客户端、连接器、分析器、优化器、执行器、查询缓存、存储引擎）
        • MySQL 8.0版本开始，查询缓存功能被完全移除了
    • 讲一下SQL语句在 MySQL 中的执行过程。
      • 在MySQL中，SQL语句的执行过程包括：客户端发送一条SQL给服务器，服务器先检查查询缓存，如果命中则直接返回结果；否则进行SQL解析、预处理，然后优化器生成执行计划，最后由存储引擎执行并返回结果。在执行过程中，涉及权限检查和查询优化等步骤。
    • MySQL提供了哪些存储引擎。
      • MySQL提供了多种存储引擎，不同版本的MySQL支持的存储引擎可能有所不同，其中InnoDB和MyISAM是最常用的。
        • MySQL 5.5.5 之前，MyISAM 是 MySQL 的默认存储引擎。5.5.5 版本之后，InnoDB 是 MySQL 的默认存储引擎。
        • MySQL 的 存储引擎有：InnoDB、MyISAM、Memory（HEAP）、MERGE、BDB（BerkeleyDB）、EXAMPLE、NDB（MySQL Cluster）、ARCHIVE、CSV、BLACKHOLE、FEDERATED 等 （了解）
    • 讲一下 MySQL中存储引擎的 InnoDB 和 MyIsam 的区别。
      • MyISAM 不支持事务和行级锁，最大的缺陷是崩溃后无法安全恢复。5.5.5 版本之后，InnoDB 是 MySQL 的默认存储引擎。
      • 关于之间的一个差别，可以从是否支持行级锁、是否支持事务、是否支持外键、是否支持数据库异常崩溃后的安全恢复、是否支持MVCC、索引实现、性能等方面来看。首先是：
      • 锁机制：InnoDB支持行级锁，提高了多用户并发操作的性能；而 MyISAM只支持表级锁。
      • 事务：InnoDB支持事务，具有提交、回滚和崩溃恢复能力，而MyISAM不支持事务
      • MVCC: MyISAM 不支持 MVCC，而 InnoDB 支持
      • 安全恢复：MyISAM 不支持，而 InnoDB 支持。使用 InnoDB 的数据库在异常崩溃后，数据库重新启动的时候会保证数据库恢复到崩溃前的状态。这个恢复的过程依赖于 redo log
      • 外键：InnoDB支持外键，可以用来维护数据的完整性；MyISAM不支持外键。不过一般不建议使用外键。在阿里的《Java 开发手册》也是明确规定禁止使用外键的。
      • 索引实现： 虽然 MyISAM 引擎和 InnoDB 引擎都是使用 B+Tree 作为索引结构，但是两者的实现方式不太一样。
        • InnoDB 引擎中，其数据文件本身就是索引文件。其表数据文件本身就是按 B+Tree 组织的一个索引结构，树的叶节点 data 域保存了完整的数据记录。
        • 而在 MyISAM 引擎，索引文件和数据文件是分离的，索引文件仅存储数据记录的地址，查找数据的话需要每次去先查找索引文件中对应数据的地址。
      • 性能： InnoDB 的性能比 MyISAM 更强大。
  • 事务
    • 关键词：事务特性、隔离级别、并发事务问题，默认隔离级别、MVCC。
    • 什么是MySQL事务。
      • MySQL 事务是由一系列的数据库操作组成，他可以保证批量操作的完整性，要么全部执行，要么全部不执行。
      • 同时事务支持原子性、隔离性、持久性和一致性的特点。
    • 事务中 的 ACID 是指什么。
      • ACID是事务必须满足的四个基本特性：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。这些特性确保数据库事务是可靠和安全的。
    • 并发事务会带来什么问题。
      • 并发事务可能导致脏读、不可重复读和幻读等问题，这些问题影响数据的一致性和完整性。数据库通过事务隔离级别来控制并发事务中数据的可见性，以解决这些问题。
    • SQL 事务隔离级别有哪些。
      • 事务隔离级别定义了一个事务可能受其他并发事务影响的程度，不同的隔离级别会导致不同的并发事务问题。
      • SQL标准定义了四种事务隔离级别：读未提交（Read Uncommitted）、读已提交（Read Committed）、可重复读（Repeatable Read）和串行化（Serializable）
    • 不可重复读和幻读的区别。
      • 不可重复读指在同一事务内，多次读取同一数据集合时，由于其他事务的修改操作，导致后续读取的数据与前面的不一致。幻读指在同一事务内，两次查询同一范围的数据时，第二次查询出现了第一次查询未出现的数据，通常是因为其他事务插入了新的数据。
    • MySQL的默认隔离是什么，能解决幻读问题吗。
      • MySQL的默认隔离级别是可重复读（Repeatable Read）。在可重复读隔离级别下，MySQL使用Next-Key Locking避免幻读问题，因此能够有效解决幻读问题。
    • InnoDB 事务隔离级别实现原理。
      • InnoDB通过锁和多版本并发控制（MVCC）来实现不同的事务隔离级别。锁机制包括行锁和表锁，用于实现串行化；MVCC允许在同一时间内，不同的事务看到不同版本的快照数据，用于实现读未提交、读已提交和可重复读。
    • 什么是MVCC，有什么用，原理了解吗。
      • MVCC，即多版本并发控制，是一种用于实现数据库事务隔离的技术。
      • MVCC允许事务在不加锁的情况下读取数据的旧版本，从而提高并发性能。它通过为每个写操作创建数据的新版本，每个读操作可以访问版本链中的某个版本，这样就能根据事务的隔离级别返回适当的数据版本。
  • 锁
    • 表级锁和行级锁了解吗，有什么区别。
      • 表级锁锁定整个表，简单、开销小，但并发度低；行级锁锁定单个或多个行，开销大，但并发度高。
      • 表级锁适合读多写少的场景，而行级锁适合高并发的读写操作。
    • 行级锁的使用有什么注意事项。
      • 行级锁使用时要注意避免死锁，尽量减少锁定时间，合理设计索引以避免间隙锁（Gap Lock），并确保查询能够利用索引，避免全表扫描引起的隐式锁定。
    • 讲一下共享锁和排他锁。
      • 共享锁（S锁）允许事务读取一行数据，多个事务可以同时持有同一资源的共享锁；
      • 排他锁（X锁）允许事务更新或删除一行数据，如果事务T对数据A加上排他锁，则只允许T读写该数据，阻止其他事务获取A的任何锁。
    • 意向锁有什么作用。
      • 意向锁是一种表级锁，用于声明事务打算在表的行上加什么类型的锁（共享或排他）。它的作用是在获取行锁之前先检查表级别的意向锁，从而避免检查表中每一行锁的状态，提高锁定的效率。
    • MySQL 的 InnoDB 引擎中行锁有几种类型？、
      • InnoDB 支持三种类型的行锁：记录锁（锁定单个行记录）、间隙锁（锁定一个范围，但不包括记录本身）、临键锁（Next-Key Lock，锁定一个范围并包括记录本身）。临键锁是默认的行锁类型，它结合了记录锁和间隙锁的特性。
  • 索引
    • 索引有什么用？讲一下索引的优缺点。
      • 索引用于提高数据库查询效率，使数据检索更快。优点：加速查询速度，提高数据检索效率，帮助数据库服务器避免全表扫描，加快排序和分组操作。缺点：占用更多的物理空间，降低更新表（INSERT、UPDATE、DELETE）的速度，因为索引也需要被更新。
    • 讲一下索引的底层结构。
      • 大多数数据库系统中，索引的底层结构主要是B-Tree或B+Tree结构。B+Tree是B-Tree的变种，具有所有数据指针都存在于叶子节点且叶子节点通过指针连接的特性，这有助于范围查询
    • MySQL 索引：索引为什么使用 B+树？
      • B+树用于索引因为它保持数据平衡，且查询时间复杂度为O(log n)，适合频繁的插入、删除和查询操作。B+树的叶子节点连续且只在叶子节点存储数据，这使得范围查询更加高效。
    • 讲一下这些区别：主键索引和二级索引、聚集索引和非聚集索引。
      • 主键索引：唯一标识数据库表中每行数据的索引。每个表只能有一个主键索引。
      • 二级索引（辅助索引）：除主键索引外的索引，可以有多个。
      • 聚集索引：按照每行数据的键值顺序存储数据，InnoDB的表数据文件本身就是按B+Tree组织的聚集索引。
      • 非聚集索引：索引顺序和物理存储顺序不同，非聚集索引指向聚集索引的指针。
    • 讲一下 联合索引、覆盖索引。
      • 联合索引：基于多个列的索引，可以提高多列查询的效率，遵循最左前缀匹配原则。
      • 覆盖索引：查询列直接在索引中，无需回表查询数据，可以显著提高查询效率。
    • 建立索引的时候会遵守最左前缀匹配原则，讲一下这个。
      • 最左前缀匹配原则指的是MySQL在使用联合索引时，会从索引的最左边开始匹配列。只有按照索引的最左列开始的查询才能利用索引，不按顺序或中间有列缺失的查询不能完全利用索引。
    • 创建索引需要注意什么。
      • 避免在经常更新的列上创建索引，以减少维护索引的成本。
      • 对于查询频繁的列创建索引，特别是作为WHERE条件的列。
      • 考虑使用联合索引来优化查询效率，但注意最左前缀原则。
      • 不要过度创建索引，以免增加存储开销和降低更新性能。
      • 考虑索引的选择性，高选择性的列（唯一值多）更适合创建索引。
  • 日志
    • MySQL 日志：常见的日志都有什么用？
      • MySQL 主要包括错误日志、查询日志、慢查询日志、二进制日志（binlog）、重做日志（redolog）、撤销日志（undo log）。它们用于错误追踪、查询记录、性能优化、数据复制、数据恢复和事务管理
    • 慢查询日志有什么用。
      • 慢查询日志记录执行时间超过指定阈值的SQL语句。它有助于识别和优化数据库性能瓶颈，通过分析和调整那些执行缓慢的查询来提高数据库的整体性能。
    • binlog 是什么，它主要记录了什么，有什么用。
      • 二进制日志（binlog）记录了所有对数据库修改的操作（如SQL语句），但不包括SELECT和SHOW这类的操作。它主要用于复制和恢复数据（数据恢复、主从复制场景）。
    • 讲一下 redolog ；同时说一下 redo log 如何保证事务的持久性。
      • 重做日志（redolog）是InnoDB存储引擎特有的日志，用于记录事务操作的物理修改。它确保事务的持久性通过在事务提交前将修改记录到持久存储，即使数据库发生故障，也能通过redolog恢复数据。
    • binlog 和 redolog 有什么区别。
      • binlog是MySQL的二进制日志，记录了对数据库的修改操作，主要用于复制和数据恢复。redolog是InnoDB存储引擎的重做日志，记录了数据页的物理修改，主要用于事务的持久性和崩溃恢复。binlog在事务提交后写入，而redolog在事务进行时就写入。
    • undo log 如何保证事务的原子性。
      • 撤销日志（undo log）记录了事务执行中对数据的修改前的状态。如果事务需要回滚，它可以利用undo log来恢复数据到事务开始前的状态，从而保证事务的原子性，即事务的操作要么全部执行，要么全部不执行。
  • 内存
    • 页修改之后为什么不直接刷盘。
      • 页修改后不直接刷盘是为了提高数据库性能和减少磁盘I/O操作。直接刷盘会造成大量的磁盘写操作，严重影响数据库的性能。通过使用缓冲池技术，修改操作首先在内存中进行，然后异步批量刷新到磁盘，这样可以合并多个写操作，减少对磁盘的访问次数，提高效率。
  • SQL优化
    • 讲一下SQL优化常见手段。
      • SQL优化的常见手段包括：1) 选择合适的索引，避免全表扫描；2) 优化查询语句，减少不必要的数据处理；3) 使用连接（JOIN）代替子查询；4) 限制数据返回量，使用LIMIT分页查询；5) 分析并优化执行计划；6) 适当使用查询缓存；7) 避免使用复杂的SQL函数和表达式；8) 调整数据库配置，优化数据库服务器性能
    • 讲一下慢查询问题排查。
      • 排查慢查询问题通常包括以下步骤：
        1. 开启慢查询日志，设置合理的阈值；
        1. 使用 EXPLAIN 分析查询执行计划，查找性能瓶颈；
        1. 检查查询中使用的索引是否合理，考虑添加或优化索引；
        1. 优化查询语句，减少不必要的数据处理；
        1. 分析数据库服务器的性能指标，如CPU、内存和I/O使用情况；
        1. 考虑调整数据库配置或硬件升级以提高性能。
