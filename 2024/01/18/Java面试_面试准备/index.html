通过求职为导向学习：

    1. 根据你想要的工作列一份技能清单
  • 2.根据技能清单写好最终的简历
    1. 按照简历的要求去学习和提示

个人总结

  • 1、准备好一份比较OK的简历，最好有相关的技术解决流程和实际场景准备
  • 2、个人介绍的时候最好讲一下自己的一个业务链和在这份工作中的个人职责，让面试官对你有一个基本印象，对于你做了什么有一个初步印象，同时说一下自己对于这家公司的一些打算，以及表现出自己的乐趣和兴趣；同时可以说一下自己在项目中遇到的一些问题和当时的解决思路和方案
  • 3、如果对方问问题，最好带入整个体系（一方面是广度，另一方面是深度），最后再以实际代码场景题为结束，将面试官当作一个不是太懂的小白，你讲述整个体系给对方听，当第二次讨论的时候，你把他当作交流的同学，讨论某个点的方案；当第三次讨论的时候，你可以说一下你的想法，然后问一下对方是怎么思考的；
  • 4、最后，最好将对方当作很不错的朋友，觉得这次交流很不错，很感谢对方和自己的交流，受益匪浅，同时跟对方讨论一下相关的工作岗位和内容，以及听一下对方的一些内容。

真正学到东西的话，建议不光要把项目单纯完成跑起来，还要去自己尝试着优化！

可以在你原有的项目基础上，进行实际功能的优化并运行。

      1. 体现你的架构思维
      1. 体现你的团队协作能力

  • 比如独立从 0 到 1 构建服务架构，第一次参与线上问题的诊断和解决

目前的问题

在准备项目描述这里，感觉会遇到一些问题：

  • 项目的背景（这个项目解决什么问题），1分钟以内要描述完
  • 项目包含的功能/模块，简要说明每个功能的作用
  • 你参与并负责的模块
  • 项目采用的技术架构；

目前遇到的几个问题：

  • 1、模块业务调用链并不是太复杂，没有涉及到太多考虑的代码层面的点，更多的是一个API调用的工作（缺乏竞争力，如果改善这步）
  • 2、打算主要围绕一个比较复杂的模块和功能去讲述，然后涉及到一些并发情况/微服务情况/中间件需要考虑的点，然后引入对于 Redis、MQ的相关问题
  • 3、对于 SpringCloud 的使用优势体现并不是很明显，相关的功能：比如分布式事务、API网关、负载均衡这些模块业务上并没有体现出来，也没有体现出对于相关的一个认识和了解
  • 4、如何体现自己的专业行和真正精通相关的技术栈（在业务功能实现上应该要讲述出来，并且要体现一定难点）
  • 5、打算体现一下设计模式相关的设计思路（包括组合模式，策略模式，工厂模式，模板模式）

要结合一下实际项目，比如说谈一下 Redis （结合实际场景，带入后续的话题）：

  • 我看你做的项目中，都用到了 Redis ，你在最近的项目中哪些场景使用了 redis 呢。
  • 扩展：一是验证项目真实性，二是作为深入发问的切入点

针对项目方面的一些准备：

数据监控

做这个项目的目的是为了监控在多个服务器上的一些JVM，内存，磁盘，日志等；

在开发无纸化系统的时候，将这套监控体系进行开发并投入到测试使用，在运行一段时间后，在其他系统：包括MES系统、无纸化系统进行了部署实施运行。

这套项目的设计是通过了 Telegraf 工具将数据采集，并放入到了 InfluxDB ，之后使用 Grafana 进行了可视化展示。

选用这套系统的目的：一方面是为了监控服务器和服务的运行状态，另一方面是这套系统的运行不影响其他服务，还有一个考虑是 Telegraf + InfluxDB + Grafana 是比较稳定的一套解决方案，开发成本较低。

还有一个比较占据比较大的因素是：这套方案的易用性和快速部署。

思考：

一般我们进行监控系统的运行有什么方式，为什么不选用其他的方式，怎么考量的。

在选择监控方案时，考虑了多种其他的监控工具和平台，如Nagios、Zabbix、Prometheus等。每种方案都有其特点和适用场景，比如Prometheus在Kubernetes环境中的集成优势显著。然而，基于以下几点考虑，最终选择了Telegraf + InfluxDB + Grafana：

  • 易用性和快速部署：相较于需要复杂配置的监控系统，Telegraf的插件化架构使得数据收集变得更加灵活和简单，而InfluxDB和Grafana的组合也大大简化了数据存储和展示的流程。
  • 专注于时序数据：InfluxDB对时序数据的优化超过了其他通用型数据库，对于监控系统这种高频写入、时序查询的场景更加适合。
  • 社区和生态支持：这套方案背后有着活跃的社区和丰富的文档资源，便于解决开发过程中遇到的问题，也便于系统的后续扩展和维护。

谈对一些线上问题如何解决，尽量从接口方面，OOM 方面等内容，运维事故等方面结合讲一下。

补充一些基础的概念：

常见的性能指标

  • QPS
    • QPS 是“每秒查询率”(Queries Per Second)的缩写，表示系统每秒可以处理的查询次数或请求次数。它是衡量Web应用程序性能和负载能力的一个关键指标，用于评估服务器或应用程序处理请求的能力。
  • RT (Response Time)
    • 响应时间，指从发送请求到收到响应所需的时间。通常用毫秒(ms)作为单位。低响应时间意味着用户可以更快地得到服务

考虑一个问题， 目前项目中是消息中间件是 RocketMQ，为什么选择使用这个消息中间件

  • 1、比较熟悉这个消息中间件的使用
  • 2、业务需要保证消息不丢失、事务处理较为复杂。
