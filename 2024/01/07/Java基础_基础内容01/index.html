1.2、Java SE vs Java EE

Java SE （Java Platform，Standard Edition） 是 Java 的基础版本，Java EE （Java Platform, Enterprise Edition ）是 Java 的高级版本。

基本介绍：

Java SE是Java编程语言的标准版，它提供了Java语言的核心功能，包括：

  • 基本语言特性：如变量、数组、循环、条件语句等。
  • 核心类库：提供了基础的类库，包括数据类型、数学函数、文件处理、网络编程、并发编程等。
  • 图形用户界面（GUI）: 如AWT（Abstract Window Toolkit）和Swing。
  • Java虚拟机（JVM）: Java程序的运行环境。
  • 工具: 如编译器（javac），解释器（java），打包工具（jar）等。

Java EE是Java的企业版，它在Java SE的基础上增加了用于构建大型企业级应用的功能。Java EE包括：

  • 扩展的类库: 在Java SE的基础上提供了额外的类库和API，用于企业级应用开发。
  • 服务器端技术: 如Servlets、JavaServer Pages（JSP）、Enterprise JavaBeans（EJB）等。
  • Web服务: 支持创建和消费Web服务。
  • 持久化API: 如Java Persistence API（JPA）。
  • 分布式计算: 支持构建大型、分布式的多层应用。
  • 集成技术: 如JMS（Java Message Service）和JavaMail。

总结

  • Java SE 提供了Java语言的基础，更适合开发桌面应用程序或简单的服务器应用程序
  • Java EE 在Java SE的基础上增加了企业级特性，更适合开发复杂的企业级应用程序或 Web 应用程序

1.5、编译与解释

高级编程语言按照程序的执行方式分为两种：

  • 编译型：编译型语言open in new window 会通过编译器open in new window将源代码一次性翻译成可被该平台执行的机器码。
    • 一般情况下，编译语言的执行速度比较快，开发效率比较低。常见的编译性语言有 C、C++、Go、Rust 等等。
  • 解释型：解释型语言open in new window会通过解释器open in new window一句一句的将代码解释（interpret）为机器代码后再执行。
    • 解释型语言开发效率比较快，执行速度比较慢。常见的解释性语言有 Python、JavaScript、PHP 等等

为什么说 Java 语言“编译与解释并存”？ → 先编译 后解释

这是因为 Java 语言既具有编译型语言的特征，也具有解释型语言的特征。

因为 Java 程序要经过先编译，后解释两个步骤，由 Java 编写的程序需要先经过编译步骤，生成字节码（.class 文件），这种字节码必须由 Java 解释器来解释执行。

1.7、Oracle JDK vs OpenJDK

JDK 原来是 SUN 公司开发的，后来 SUN 公司又卖给了 Oracle 公司; 原来的 SUN 公司把 JDK 给开源了； Oracle 公司就类似开了一个分支，做了一个闭源版本。

区别：

OpenJDK

  1. 开源：OpenJDK是一个开放源代码的Java SE平台实现，它是自由软件并且公开发布。它是Oracle JDK的基础。
  2. 社区驱动：由开源社区维护和更新。它通常是最先获得最新语言特性和修复的版本。
  3. 许可证：使用GPL（通用公共许可证）v2许可证，带有Classpath例外，允许不受GPL限制地使用类库。
  4. 更新：通常更加频繁地获得更新和补丁。
  5. 广泛使用：由于其开源性质，被广泛用于各种环境，包括商业应用。

Oracle JDK

  1. Oracle官方版本：Oracle JDK是由Oracle公司提供的Java SE平台的官方实现。它基于OpenJDK，并包含了一些附加组件。
  2. 商业用途许可费：Oracle JDK的早期版本免费用于个人和开发用途，但从JDK 11开始，Oracle更改了许可策略，对于商业用途或生产环境中的使用需要付费。
  3. 性能优化和附加特性：Oracle JDK可能包括一些专有的性能增强和其他特性，如JRockit或Mission Control。
  4. 支持：Oracle提供专业的长期支持服务，适合企业用户。
  5. 更新频率：Oracle JDK的更新可能不如OpenJDK频繁，但更稳定，经过了更严格的测试。

总结

  • OpenJDK 是一个完全开源的JDK版本，适用于需要免费、开源解决方案的场景。
  • Oracle JDK 提供了一些附加特性和专业支持，适用于需要稳定支持和企业级特性的商业应用。

  • Java 不提供指针来直接访问内存，程序内存更加安全
  • Java 的类是单继承的，C++ 支持多重继承；虽然 Java 的类不可以多继承，但是接口可以多继承。

2.3 自增自减运算符

符号在前就先加/减，符号在后就后加/减

Java中的自增（++）和自减（--）运算符是常用的一元运算符，用于增加或减少变量的值。

自增运算符（++）

自增运算符++用于将变量的值增加1。它有两种形式：

  1. 前缀自增：++var。在表达式求值之前增加变量的值。

    • 例如：
    • int a = 5; int b = ++a; // 先将a增加1，变成6，然后将a的值赋给b，b也是6

  2. 后缀自增：var++。在表达式求值之后增加变量的值。

    • 例如：
    • int a = 5; int b = a++; // 先将a的值赋给b，b是5，然后a增加1，变成6

自减运算符（--）

自减运算符--用于将变量的值减少1。它同样有两种形式：

  1. 前缀自减：--var。在表达式求值之前减少变量的值。

    • 例如：
    • int a = 5; int b = --a; // 先将a减少1，变成4，然后将a的值赋给b，b也是4

  2. 后缀自减：var--。在表达式求值之后减少变量的值。

    • 例如：
    • int a = 5; int b = a--; // 先将a的值赋给b，b是5，然后a减少1，变成4

使用场景和注意事项

  • 这些运算符通常用在循环和简单的数值操作中。
  • 选择前缀还是后缀形式取决于你希望在表达式其他部分使用变量的哪个值（增加或减少前的值，还是增加或减少后的值）。
  • 在单独使用时（不作为更大表达式的一部分），前缀和后缀形式的效果是相同的。

2.4 移位运算符

  • << :左移运算符，向左移若干位，高位丢弃，低位补零。x << 1,相当于 x 乘以 2(不溢出的情况下)。
  • >> :带符号右移，向右移若干位，高位补符号位，低位丢弃。正数高位补 0,负数高位补 1。x >> 1,相当于 x 除以 2。
  • >>> :无符号右移，忽略符号位，空位都以 0 补齐。

2.6 switch case

语法

switch case用来判断一个变量与一系列值中某个值是否相等，每个值称为一个分支

使用规则：

  • switch语句中变量类型可以是：byte、short、int、char。从Java SE 7开始，switch支持String类型，同时case必须为字符串常量
  • switch语句可以拥有多个case，每个case后面跟一个要比较的值和冒号
  • case语句中的值的数据类型必须与变量的数据类型相同
  • 当变量的值与case语句的值相等，case语句之后的语句开始执行，直到出现break语句才会跳出switch
  • 当遇到break时，switch终止。程序跳转到switch后面的语句执行。case不必须要包含break。如果没有break出现，程序会继续执行下一条case，直到出现break
  • switch可以包含一个default分支，该分支是switch的最后一个分支，该分支可以在任何位置，但建议写在最后。default在没有case的值和变量相等时执行。default不需要break

switch case 支持哪几种数据类型

在Java中，switch语句支持以下几种数据类型：

  1. 整数类型:
    • 包括byte、short、int以及它们的包装类Byte、Short、Integer。
    • char和Character也被支持，因为它们可以被转换为整数类型。
  2. 枚举类型 (enum):
    • 可以使用枚举类型的常量作为case标签。
  3. 字符串类型 (String):
    • 从Java 7开始，switch语句支持字符串类型。
    • 字符串比较是基于字符串的内容而不是引用。
  4. Java 12 引入的新特性:
    • 从Java 12开始，switch表达式支持更多的数据类型和更灵活的使用方式（作为预览特性）。

需要注意的是，long、float、double及其包装类不被switch语句支持。此外，switch语句中的case标签必须使用常量表达式，不能使用变量。

switch语句的这种类型限制主要是出于性能考虑，因为switch通常是通过整数跳转表实现的，这使得switch语句在处理整数类型时非常高效。使用枚举和字符串则是为了提供更多的编程便利。

3.2 基本类型和包装类型

区别

  • 用途：除了定义一些常量和局部变量之外，我们在其他地方比如方法参数、对象属性中很少会使用基本类型来定义变量。并且，包装类型可用于泛型，而基本类型不可以。
  • 存储方式：基本数据类型的局部变量存放在 Java 虚拟机栈中的局部变量表中，基本数据类型的成员变量（未被 static 修饰 ）存放在 Java 虚拟机的堆中。包装类型属于对象类型，我们知道几乎所有对象实例都存在于堆中。
  • 占用空间：相比于包装类型（对象类型）， 基本数据类型占用的空间往往非常小。
  • 默认值：成员变量包装类型不赋值就是 null ，而基本类型有默认值且不是 null。
  • 比较方式：

特点

  1. 对象特性：包装类型是对象，可以调用各种方法，例如Integer.parseInt()。
  2. 可用于泛型：在集合类等使用泛型的场合，只能使用对象，因此必须使用包装类型。
  3. 可表示空值：可以表示为null，而基本类型不能。
  4. 堆内存：对象通常存储在堆上，这可能导致处理稍慢于基本类型。
  5. 自动装箱和拆箱：Java提供自动将基本类型转换为包装类型（装箱）和将包装类型转换为基本类型（拆箱）的功能。

比较和选择

  • 使用基本数据类型时，如果关注性能和内存，并且不需要对象的特性，比如在大量数值计算中。
  • 使用包装类型时，如果需要对象的特性，比如在使用集合类、需要表示空值或使用泛型时。

3.3 包装类型的缓存机制

Java中的某些包装类实现了一个称为“缓存机制”的特性，主要是为了提高性能和减少内存使用。这一机制最常见于Integer和其他几种数字类型的包装类中。我们来详细探讨这一点。

包装类型缓存机制的概述

  1. Integer缓存: Java的Integer类缓存了从-128到127之间的整数。这意味着在这个范围内的整数会被预先创建并缓存。当代码需要这些值的包装实例时，会重用已经创建的对象，而不是每次都创建新对象。
  2. 其他类型: 类似的缓存机制也适用于Byte, Short, Long, Character等包装类。但具体的缓存范围可能不同。例如，Byte缓存了所有可能的值（从-128到127），而Character缓存了从\u0000到\u007F的字符。

缓存机制的工作原理

当你通过valueOf方法（这是自动装箱背后使用的方法）或者直接使用自动装箱来创建一个包装对象时，Java会首先检查其值是否在缓存范围内。如果是，它会从缓存中返回相应的对象实例；如果不是，它会创建一个新的对象实例。

缓存机制的好处

  • 性能提升：通过减少不必要的对象创建，提高了性能。
  • 内存优化：减少了内存的占用，因为共享了相同的对象。

使用时的注意事项

  • 当比较包装类型的值时，应该使用equals方法而不是== 运算符，因为 == 比较的是引用，而不是值。但在缓存范围内，使用 == 可能会误导你认为比较的是值。
  • 在处理大量的整数运算时，要意识到自动装箱和拆箱可能会引入额外的性能开销

Java 基本数据类型的包装类型的大部分都用到了缓存机制来提升性能。

Byte,Short,Integer,Long 这 4 种包装类默认创建了数值 [-128，127] 的相应类型的缓存数据，

Character 创建了数值在 [0,127] 范围的缓存数据，

Boolean 直接返回 True or False。

Java中的包装类型（如Integer、Long、Byte、Character、Short等）使用了常量池技术，主要目的是为了减少不必要的对象创建，提高空间和时间效率。

示例：

常量池的一些相关内容：

  1. 常量池:
    • 常量池是一种特殊的存储区域，用于存储包装类的常用对象。
    • 在Java中，Byte、Short、Integer、Long、Character这些类的对象有常量池。
  2. Integer常量池范围:
    • 对于Integer，常量池的范围默认是从-128到127。
    • 当你创建一个在这个范围内的Integer对象时，Java会直接从常量池中返回已存在的对象而不是创建一个新的对象。
  3. 超出常量池范围:
    • 如果创建的对象超出了常量池的范围，Java会创建一个新的对象。
  4. 节省内存:
    • 由于常量池的使用，相同的数值会指向相同的对象，这在一定程度上节省了内存。
  5. Character常量池范围:
    • 对于Character，常量池的范围是从\u0000到\u007F（即0到127）。
  6. 自动装箱:
    • 当你将基本数据类型转换为包装类时，Java会自动使用常量池（如果值在常量池的范围内），这个过程称为自动装箱。
  7. 注意事项:
    • 在比较包装类型对象时，应该使用equals方法而不是== ，因为 == 比较的是引用，而不是值。但在常量池范围内， == 可以正常工作因为引用指向相同的对象。

常量池技术是Java中一种重要的内存优化手段，它在处理大量的包装类型对象时尤为有效。

3.4 自动装箱与拆箱

  • 装箱：将基本类型用它们对应的引用类型包装起来；
  • 拆箱：将包装类型转换为基本数据类型；

实现：

自动装箱（Autoboxing）:

  • 当需要一个包装类型的对象时，Java会自动将基本数据类型转换为对应的包装类型。
  • 这是通过调用包装类的静态valueOf方法实现的。
  • 例如，当你写Integer boxed = 10;时，实际上执行的是Integer boxed = Integer.valueOf(10);。

自动拆箱（Unboxing）:

  • 当需要一个基本数据类型的值时，Java会自动将包装类型的对象转换为基本数据类型。
  • 这是通过调用包装类对象的xxxValue方法实现的（例如，intValue、doubleValue等）。
  • 例如，当你写int unboxed = boxed;时，实际上执行的是int unboxed = boxed.intValue();。

注意事项:

  • 空指针异常：自动拆箱可能导致空指针异常。如果包装类型的引用为null，在自动拆箱时会尝试调用null的xxxValue方法，从而抛出NullPointerException。
  • 性能考虑：虽然自动装箱和拆箱很方便，但在性能敏感的场景下应当谨慎使用，因为装箱和拆箱操作涉及额外的对象创建和方法调用。

自动装箱和拆箱让开发者可以更加自然地编写代码，无需显式地在基本数据类型和包装类型之间转换，但同时也需要了解其背后的原理和潜在陷阱。

可能导致NPE的自动拆箱

NPE 指的是空指针异常

上述我们也讲到，如果包装类型的引用为null，在自动拆箱时会尝试调用null的xxxValue方法，从而抛出NullPointerException。

说一下这种情况如何处理：

处理方法:

  1. 检查空值:
    • 在进行自动拆箱之前，先检查包装类型的对象是否为null。
    • 示例：if (myInteger != null) { myInt = myInteger; }
  2. 使用默认值:
    • 如果包装类型可能为null，可以使用默认值来避免NPE。
    • 示例：myInt = (myInteger != null) ? myInteger : defaultValue;
  3. 避免不必要的装箱和拆箱:
    • 在可能的情况下，尽量避免不必要的自动装箱和拆箱，特别是在关键的业务逻辑中。
  4. 使用Optional类:
    • 在Java 8及以上版本，可以使用Optional类来优雅地处理可能为null的情况。
    • 示例：myInt = Optional.ofNullable(myInteger).orElse(defaultValue);
  5. 代码审查和测试:
    • 在编码阶段进行严格的代码审查，确保对可能为null的对象进行了适当的处理。
    • 编写单元测试来覆盖可能产生NPE的场景。

原因分析：

浮点数运算中的精度丢失是一个在计算机科学中广泛存在的问题，尤其是在使用单精度（float）和双精度（double）类型时。这个问题的根源在于浮点数在计算机中的表示方式。

浮点数表示

浮点数在计算机中通常使用IEEE 754标准表示。这种表示方法有两个主要部分：

  1. 尾数（Mantissa）：代表数字的有效数字。
  2. 指数（Exponent）：决定小数点的位置。

由于这两个部分都有固定的位数，所以只能精确表示一定范围和精度的数。一些数，如分数1/3或特定的小数，无法被精确地表示为浮点数。

精度丢失的原因

  1. 有限的位数：由于尾数和指数的位数是有限的，因此浮点数不能精确表示所有的数值。在转换和运算过程中，会进行四舍五入以适应这些位数，从而导致精度丢失。
  2. 二进制表示：某些十进制小数在二进制下是无限循环小数，无法精确表示，例如0.1。

实际影响

在进行浮点数运算时，如加法、减法、乘法和除法，这种精度的丢失可能会累积，导致最终结果与预期有较大偏差。这在金融计算等需要高精度的场合尤为重要。

解决方法

  1. 使用BigDecimal: 在Java中，BigDecimal类可以用于精确的小数运算。它允许控制舍入模式，可以用于金融和科学计算中。
  2. 避免不必要的运算：减少浮点数运算的数量可以减少累积误差。
  3. 理解和接受一定的误差：在某些场合，完全精确的结果可能并不是必需的，此时理解并接受浮点运算的限制是合理的。

四、变量

4.1 成员变量与局部变量

成员变量（字段、属性），局部变量在声明位置、作用域、初始化行为等方面有显著的不同

成员变量（Member Variables）

成员变量是定义在类中的变量，它们描述了类的属性或状态。根据是否使用static关键字，成员变量可以进一步分为实例变量和类变量。

  1. 实例变量：没有用static修饰的成员变量。每个类的实例都有自己的一套实例变量的副本。
  2. 类变量（静态变量）：用static修饰的成员变量。它们属于类本身，所有实例共享同一个静态变量的副本。

特点

  • 作用域：整个类内部。
  • 生命周期：与对象的生命周期（实例变量）或类的生命周期（静态变量）相同。
  • 默认值：自动初始化为默认值（如int为0，对象引用为null）。
  • 访问方式：可以通过对象（实例变量）或类名（静态变量）访问。

局部变量（Local Variables）

局部变量是在方法内、构造器内或代码块内定义的变量。

特点

  • 作用域：仅限于它们被声明的方法、构造器或代码块内。
  • 生命周期：从声明开始到方法、构造器或代码块执行完毕时结束。
  • 初始化：不会自动初始化，必须在使用前显式初始化。
  • 内存位置：通常存储在栈上。

比较

特性 成员变量 局部变量
声明位置 类体内部 方法、构造器或代码块内
作用域 整个类内部 仅限于声明它们的块内
生命周期 实例变量：随对象存在；静态变量：随类存在 随方法、构造器或代码块执行完毕而结束
初始化 自动初始化为默认值 必须显式初始化
内存位置 堆（实例变量）或方法区（静态变量） 栈

5.3 静态方法与实例方法

区别

静态方法（Static Methods）

  1. 定义: 使用static关键字定义。
  2. 调用方式: 可以通过类名直接调用，不需要创建类的对象。
  3. 访问: 只能直接访问类的静态成员（变量和方法）。不能直接访问实例成员。
  4. 用途: 常用于不依赖于对象状态的工具或帮助方法。

实例方法（Instance Methods）

  1. 定义: 没有static关键字，属于类的实例。
  2. 调用方式: 需要创建类的对象，通过对象调用。
  3. 访问: 可以访问类的实例成员（变量和方法）以及静态成员。
  4. 用途: 通常用于操作或访问对象的实例数据。

比较

特性 静态方法 实例方法
定义 使用 static 关键字 没有 static 关键字
调用方式 通过类名调用（例如 ClassName.method()） 通过对象实例调用（例如 object.method()）
访问类的成员 只能访问静态成员 可访问实例成员和静态成员
对象状态依赖 不依赖对象状态 依赖或可访问对象状态
典型用途 工具方法、计算方法、无需对象状态的操作 操作或访问对象的实例数据

类比

  • 静态方法: 就像是大学的公共图书馆，所有学生（类的实例）都可以使用，但它不属于任何一个学生。
  • 实例方法: 就像学生的个人笔记本，每个学生都有自己的笔记本，可以在其中写下自己的笔记（实例数据），并且每个笔记本都是独一无二的。

静态方法不能调用非静态成员

主要原因如下：

  1. 静态方法是属于类的，在类加载的时候就会分配内存，可以通过类名直接访问。而非静态成员属于实例对象，只有在对象实例化之后才存在，需要通过类的实例对象去访问。
  2. 在类的非静态成员不存在的时候静态方法就已经存在了，此时调用在内存中还不存在的非静态成员，属于非法操作

5.4 重载和重写

重载：（同一个类中的操作）

发生在同一个类中（或者父类和子类之间），方法名必须相同，参数类型不同、个数不同、顺序不同，方法返回值和访问修饰符可以不同。

重写：（子类对父类的操作）

重写发生在运行期，是子类对父类的允许访问的方法的实现过程进行重新编写。

  1. 方法名、参数列表必须相同，子类方法返回值类型应比父类方法返回值类型更小或相等，抛出的异常范围小于等于父类，访问修饰符范围大于等于父类。
  2. 如果父类方法访问修饰符为 private/final/static 则子类就不能重写该方法，但是被 static 修饰的方法能够被再次声明。
  3. 构造方法无法被重写
特性 重载 (Overloading) 重写 (Overriding)
所在位置 同一个类中 子类中
方法名 必须相同 必须相同
参数列表 必须不同（数量、类型或顺序） 必须相同
返回类型 可以不同 必须与被重写的方法兼容
访问修饰符 可以不同 不能比父类方法更严格
抛出的异常 可以不同 不能抛出新的检查异常或更广泛的检查异常
实现目的 增加同一操作的灵活性 实现运行时多态，提供特定的实现
重载和重写_02

重载和重写的区别

重载（Overloading）:

  • 发生在同一个类中。
  • 指的是一个类中有多个同名方法，但它们的参数列表不同（参数的数量、类型或者顺序不同）。
  • 与方法的返回类型、访问修饰符无关。
  • 编译时多态：方法调用是根据方法签名（方法名和参数列表）在编译时确定的。

重写（Overriding）:

  • 发生在父子类之间。
  • 子类提供了一个具有相同名称、返回类型及参数列表的方法，用以重写（覆盖）父类中的相应方法。
  • 访问权限不能比父类中被重写的方法更严格。
  • 运行时多态：具体调用哪个方法是在运行时根据对象的实际类型确定的。
  • 必须保持一致的异常声明（或不声明异常）。

这两种机制是Java实现多态性的重要手段。重载允许同一个类中定义功能相似但参数不同的方法，而重写则允许子类改变或扩展从父类继承的行为.

重载与方法的返回类型、访问修饰符无关

在Java中，方法重载（Overloading）的定义是指在同一个类中可以存在多个同名方法，只要它们的参数列表不同即可。这里的“参数列表不同”可以是参数的数量、类型或者参数顺序的不同。

当提到“与方法的返回类型、访问修饰符无关”时，意思是：

  1. 返回类型无关:
    • 重载的方法可以有不同的返回类型。方法的返回类型不是重载考虑的因素。
    • 比如，你可以有一个返回void的display()方法和一个返回String的display(String message)方法。
  2. 访问修饰符无关:
    • 重载的方法可以有不同的访问权限（如public、private等）。
    • 重载仅仅关注于方法的名称和参数列表。

这意味着，当你重载一个方法时，你可以改变它的返回类型和访问修饰符，但这不会影响到重载的本质。重载完全是基于方法的参数列表的。重载提供了一种便利，使得你可以用同一个方法名来定义做相似事情但参数不同的方法。
