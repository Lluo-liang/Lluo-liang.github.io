面试解答篇

  • 基础概念
    • Redis 有什么作用，为什么要用缓存，为什么选择Redis。
      • Redis作为内存数据库，主要用于缓存数据，提高应用访问速度，减轻后端数据库压力。使用缓存可以暂存热点数据，实现快速读取。选择Redis是因为它性能高、支持丰富数据类型、具备持久化能力、支持事务、可扩展性好，以及拥有丰富的特性如消息队列、发布/订阅等。
    • Redis 除了做缓存，还能做什么，可以做消息队列吗？
      • 除了做缓存，Redis还可以用作消息队列、分布式锁、计数器、会话存储、实时分析等。它的发布/订阅模型可以实现消息队列功能，但对于需要高可靠性和消息持久化的场景，可能需要结合其他消息中间件使用。
    • 分布式缓存常见的技术选型方案有哪些。
      • 分布式缓存的常见技术选型包括Redis、Memcached、Hazelcast等。Redis支持丰富的数据类型，适用于多种场景；Memcached适合简单的键值缓存；Hazelcast提供了分布式计算能力。
    • Redis 给缓存数据设置过期时间有什么用。
      • 设置过期时间可以让数据在一定时间后自动删除，适用于那些只需要暂时存储的数据，如验证码、会话。这样可以有效管理内存，防止缓存数据无限增长导致内存溢出。
    • Redis 是如何判定数据是否过期。
      • Redis使用惰性删除和定期删除两种策略来判定数据是否过期。惰性删除在访问数据时判断是否过期，过期则删除；定期删除则是Redis定时随机检查一些键，过期则删除
    • 过期的数据的删除策略了解吗。
      • 过期数据的删除策略主要包括惰性删除和定期删除。惰性删除是在访问键时判断是否过期，过期则删除；定期删除是周期性地扫描一部分键，删除其中的过期键。这两种策略结合使用，以平衡删除操作对性能的影响。
    • Redis 单线程模型了解吗
      • Redis单线程模型指的是数据操作是单线程执行的，这样简化了数据操作的并发控制，由于Redis是基于内存的操作，单线程的性能瓶颈不是CPU，而是内存和网络带宽，因此Redis能够以较高性能运行。
    • Redis 6.0 之前为什么不使用多线程，Redis 之后为啥引入了多线程。
      • Redis 6.0之前不使用多线程是因为其设计目标是简单高效，单线程模型能够满足大多数场景的性能需求。6.0版本之后引入了多线程主要是为了更好地利用多核CPU，提高网络I/O操作的效率，多线程用于处理客户端的请求读写，而数据操作仍然是单线程，这样既提升了性能，又保持了操作的简单性。
  • 数据结构
    • Redis 常用的数据结构有哪些。
      • String、Set、Hash、List、ZSet
      • Redis支持的数据结构包括字符串（String）、列表（List）、集合（Set）、有序集合（Sorted Set，也叫ZSet）、哈希（Hash）、位图（Bitmap）、HyperLogLog、地理空间（Geo）索引等。
  • 持久化、内存
    • Redis 内存淘汰机制了解吗
      • Redis内存淘汰机制是当内存使用达到设定的上限时，根据配置的策略自动移除部分数据。常见的淘汰策略包括无淘汰、随机淘汰、最少使用（LFU）、最近最少使用（LRU）、过期时间最短等。
    • 什么是 RDB 持久化。
      • RDB（Redis DataBase）持久化是指将Redis在某一时刻的数据快照保存到磁盘的过程。这一过程可以手动触发，也可以配置为定时或达到一定条件自动执行。RDB文件是一个压缩的二进制文件，适用于灾难恢复。
    • 什么是 AOF 持久化。
      • AOF（Append Only File）持久化是指将Redis的操作命令追加到文件的末尾。AOF文件记录了使数据库达到当前状态所执行的所有写操作命令。AOF提供了更好的持久化和数据安全性，但可能会因为文件体积的增加而影响性能。
    • Redis 4.0 对于持久化机制做了什么优化？
      • Redis 4.0引入了混合持久化模式，该模式结合了RDB和AOF的优点。在AOF重写过程中，同时将内存中的数据状态以RDB格式写入AOF文件头部，之后追加新的AOF命令。这种方式优化了数据恢复的速度，同时减少了数据恢复时的内存使用。
  • 事务
    • 如何使用 Redis 事务，Redis 事务支持原子性吗，Redis 事务有什么缺陷。
      • Redis事务通过MULTI、EXEC、DISCARD和WATCH命令来使用。Redis事务支持原子性，即事务内的命令要么全部执行，要么全部不执行。缺陷在于它不支持回滚，一旦执行EXEC后，即使某些命令失败，事务中的其他命令仍然会被执行。
    • 如何解决 Redis 事务的缺陷。
      • 解决Redis事务缺陷主要依赖于应用层逻辑来实现，比如在事务执行前进行充分的数据验证，使用WATCH命令做乐观锁以避免事务执行期间数据被其他客户端修改。
      • 对于事务执行后的错误处理，需要应用程序根据错误类型进行适当的处理或重试逻辑。
  • 生产问题
    • 怎么保证 Redis 挂掉之后再重启数据可以进行恢复？
      • 可以通过配置Redis的持久化机制来保证数据恢复，包括RDB（快照保存）和AOF（命令追加）两种方式。通过定期或条件触发的RDB快照保存，或者实时的AOF日志记录，Redis重启后可以从持久化文件恢复数据。
    • 什么是 bigkey？ 有什么危害。
      • bigkey指的是数据量特别大的键，例如一个包含数百万个元素的列表或集合。bigkey在执行操作时会消耗大量的CPU和内存资源，可能导致Redis服务响应变慢，甚至影响到Redis的稳定性。
    • 如何发现 bigkey。
      • 可以使用Redis命令MEMORY USAGE key查看键占用的内存大小，或使用redis-cli --bigkeys命令扫描整个数据库中的bigkey。
    • 如何避免大量 key 集中过期。
      • 可以通过将过期时间分散开，避免在同一时间大量key同时过期。例如，为key的过期时间添加随机值，使得它们在不同时间过期，减少集中过期带来的压力。
    • 在使用Redis作为缓存时，遇到缓存穿透，你会怎么处理。
      • 处理缓存穿透可以使用空对象缓存和布隆过滤器两种策略。对于查询不到的数据，可以将一个空对象存入缓存并设置较短的过期时间；使用布隆过滤器预先过滤掉绝对不存在的数据访问请求。
    • 讲一下缓存雪崩、缓存击穿。
      • 缓存雪崩是指缓存中大量或全部数据同时过期，导致所有请求都落到数据库上，可能会使数据库压力骤增，甚至崩溃。
      • 缓存击穿是指热点key突然过期，导致这个时刻所有请求都落到数据库上，同样可能会对数据库造成较大压力。
    • 当数据同时存储在数据库和缓存中时，如何保证缓存和数据库之间的数据一致性？
      • 保证缓存和数据库间数据一致性的策略包括：
      1. 读取操作优先从缓存获取，缓存没有则读数据库，然后写入缓存。
      2. 更新操作先写数据库，成功后再删缓存或更新缓存。
      3. 使用消息队列等异步处理机制，确保数据库和缓存最终一致
    • 在分布式缓存环境下，如何处理多个节点之间的缓存一致性问题。
      • 在分布式缓存环境下处理缓存一致性问题的策略包括：
        • 使用发布/订阅模式，当一个节点更新了数据后，通过消息系统通知其他节点也对应更新或删除缓存。
        • 通过设置缓存数据的版本号，使用最新的版本号来保证数据的一致性。
        • 使用分布式锁或事务保证操作的原子性，确保多个节点操作数据的一致性。
  • 内存管理
    • Redis 的默认内存分配器是哪个？
      • Jemalloc
      • Redis的默认内存分配器是jemalloc，它被设计用来提高内存分配效率和减少内存碎片，特别是在多线程环境下。
    • Redis 在什么情况下会清理内存数据？
      • 什么时候会是否内存，一种情况是键过期，还有就是触发了内存淘汰策略的时候（在内存使用达到限制时）
    • 讲一下常用的内存管理配置。
      • 可以在 Redis 配置文件中设置 maxmemory 选项来限制 Redis 使用的最大内存量。
      • 还可以通过 maxmemory-policy 定义当内存达到限制时 Redis 应该执行的淘汰策略。
    • Redis 内存使用监控了解吗？
      • 可以通过一些指令查看内存使用情况，比如: used_memory、used_memory_rss、used_memory_peak 等
  • 性能优化
    • 什么时候会产生 Redis 内存碎片？
      • Redis内存碎片主要在频繁的增加和删除不同大小的数据时产生。特别是当小对象被释放后，留下的空间可能无法被新的、稍大一点的对象使用，导致内存利用率下降。
    • 什么是 Redis 内存碎片，为什么会有 Redis 内存碎片？
      • Redis内存碎片是指已分配的内存中未被有效利用的部分。它主要是由于频繁地对数据进行增加、删除操作所引起的，这些操作会导致内存中出现很多小的、不连续的空闲区域。这些空闲区域虽然总量可能较大，但由于分散在各处，无法为需要较大连续内存空间的数据分配使用，从而导致内存碎片。
  • 高可用
    • 如何保证 Redis 服务高可用。
      • 保证Redis服务高可用可以通过使用哨兵（Sentinel）系统进行故障检测和自动故障转移，以及通过Redis Cluster提供分布式存储和容错功能。
    • Sentinel（哨兵）有什么作用。
      • Sentinel负责监控Redis主从服务器的运行状态，实现自动故障发现和故障转移，保证系统的高可用性。它还可以负责配置提醒和客户端重定向。
    • Redis Cluster 虚拟槽分区有什么优点。
      • 虚拟槽分区使得Redis Cluster可以在多个节点间分布数据和负载，提高了数据管理的灵活性和系统的可扩展性。这样即使某个节点失败，也只影响到该节点的一部分数据，其他节点可以继续提供服务。
    • Redis Cluster 中的各个节点是如何实现数据一致性的。
      • Redis Cluster通过异步复制、最终一致性模型来实现数据一致性。每个写操作首先在主节点执行，然后异步复制到相关的从节点。虽然这可能导致短暂的数据不一致，但通常很快就能达到一致状态。
    • 为什么要用分布式缓存。
      • 分布式缓存可以提供比单一缓存服务器更大的存储容量和更高的处理能力，以及更好的容错和可扩展性，满足大规模分布式系统中高性能和高可用性的需求。
    • 常见的缓存更新策略有哪几种?
      • 常见的缓存更新策略包括缓存穿透、缓存雪崩和缓存击穿。更新策略如LRU（最近最少使用）、TTL（生存时间）和主动更新等。
    • Redis主从复制：主从节点之间如何同步数据？
      • 在Redis主从复制中，主节点将数据的更改记录到命令日志中，从节点通过连接主节点并请求该命令日志来同步数据。从节点执行这些命令来保持与主节点的数据一致
    • Redis Sentinel：如何实现自动化地故障转移？
      • 当Sentinel监测到主节点故障时，它会自动将其中一个从节点提升为新的主节点，并通知其他从节点和客户端更新配置，实现故障转移。
    • Redis Cluster：缓存的数据量太大怎么办？
      • 当缓存数据量太大时，可以通过添加更多的节点来进行水平扩展，利用Redis Cluster的分区特性来分散数据和负载，从而处理更大的数据量。
  • 扩展问题
    • 使用 Redis 统计网站 UV 怎么做？
      • 可以使用Redis的HyperLogLog数据结构来统计UV，因为HyperLogLog提供了基于概率的去重计数功能，非常适合用来做大规模数据的去重计数，如UV统计。
    • 使用 Redis 实现一个排行榜 怎么做。
      • 可以使用Redis的有序集合（Sorted Set）数据结构来实现排行榜，通过ZADD添加成员的分数，通过ZRANK或ZREVRANK来获取成员的排名，ZSCORE获取成员的分数。
    • Redis 缓存的数据量太大怎么办。
      • 当Redis缓存的数据量太大时，可以考虑使用数据分片（Sharding）、读写分离、优化数据结构、设置合理的过期策略或使用Redis Cluster进行水平扩展等方法来解决。
