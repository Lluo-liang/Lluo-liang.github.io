面试解答篇

  • 基础内容
    • 并发和并行有什么区别。
      • 并发是多个任务在同一时刻内同时进行；
      • 并行是多个任务在同一段时间段内同时进行。
    • 什么是线程，线程和进程的区别是什么。
      • 线程是程序执行的最小单位，它是进程的一部分，共享进程的资源。进程是操作系统分配资源和调度的基本单位，每个进程至少包含一个线程。区别：进程有独立的内存空间，而线程共享进程的内存空间；进程间通信（IPC）成本较高，上下文切换较慢，线程间通信和切换成本较低；线程是实现多任务的最小单位，进程是拥有资源的最小单位。
    • 为什么要使用多线程。
      • 使用多线程可以提高程序的执行效率和响应速度。多线程允许并发执行多个任务，可以充分利用多核CPU的计算能力，提高资源利用率。在I/O密集型或者需要同时执行多个独立任务的应用场景中，多线程可以显著提高程序性能和用户体验。
    • 什么是线程死锁，如何预防和避免线程死锁。
      • 线程死锁是指两个或多个线程在执行过程中因争夺资源而造成的一种相互等待的现象，导致它们都无法继续执行。预防和避免方法：1) 避免嵌套锁；2) 按顺序加锁；3) 使用超时尝试获取锁；4) 减少锁的使用范围和持有时间；5) 使用专门的算法，如银行家算法。
    • 并发编程的三个重要特性。
      • 并发编程的三个重要特性是：1) 原子性，保证操作的不可分割；2) 可见性，保证线程对变量的修改对其他线程是可见的；3) 有序性，保证程序执行的顺序按照代码的先后顺序执行。这些特性是实现线程安全的基础。
  • 线程的基本使用
    • 线程的生命周期和状态讲一下。
      • 线程的生命周期包括以下状态：1) 新建（New）：创建后未启动。2) 可运行（Runnable）：可能正在运行或等待CPU分配时间片。3) 阻塞（Blocked）：等待监视器锁。4) 等待（Waiting）：等待其他线程执行特定动作。5) 超时等待（Timed Waiting）：在指定时间内等待。6) 终止（Terminated）：线程执行完毕或被终止。
    • 如何优雅的终止一个线程
      • 优雅地终止线程的方法包括：1) 使用标志位：设置一个标志位，线程周期性检查该标志位是否通知自己退出。2) 使用中断：调用线程的 interrupt() 方法发送中断请求，线程中断时退出循环或清理资源。
    • 讲一下线程中的乐观锁和悲观锁，并讲一下其对应的一种实现方案。（如何实现乐观锁和悲观锁）
      • 乐观锁：假设没有冲突直接进行操作，如果因为冲突失败则重试。实现方式：使用版本号或时间戳检查，如在数据库操作中，更新时比较版本号。
      • 悲观锁：假设会发生冲突，先加锁后操作。实现方式：数据库的行锁或表锁，或者在Java中使用synchronized关键字或Lock接口实现锁
    • 乐观锁和悲观锁的区别。
      • 乐观锁通常用在冲突较少的场景中，通过使用版本号或时间戳实现，不会真正加锁；而悲观锁假设冲突经常发生，通常通过数据库锁或Java中的同步机制来实现，会真正阻塞其他线程的访问。
    • 如何实现乐观锁。
      • 乐观锁可以通过版本号机制实现，每次数据更新时，同时更新版本号。更新操作首先检查当前版本号是否与读取时的版本号一致，如果一致则进行更新，否则操作失败。这种方式常用于数据库操作
    • 乐观锁存在哪些问题。
      • 乐观锁的问题包括：1) 冲突多时，回滚重试可能导致性能问题；2) 如果不正确处理，可能导致更新丢失；3) 在分布式系统中，时间同步问题可能导致乐观锁失效
  • 内存模型
    • 什么是 JMM, 为什么需要 JMM？
      • Java内存模型（JMM）是一种规范，用来定义线程和主内存之间的交互规则。JMM解决了可见性、原子性、有序性问题，确保在多线程环境下的一致性和线程安全性。需要JMM是因为不同的硬件平台有不同的内存访问模式，JMM为Java程序在各平台之间提供一致的内存可见性保证。
    • JMM 是如何抽象线程 和 主内存之间的关系。
      • JMM把内存分为主内存（所有线程共享）和工作内存（每个线程私有）。线程对变量的所有操作（读取、赋值）必须在工作内存中进行，然后再同步回主内存。JMM通过这种抽象确保了线程间通过主内存进行通信，同时保证了内存的可见性和有序性。
    • Java 内存区域和 JMM 有何区别。
      • Java内存区域（如堆、栈、方法区）涉及到数据存储的物理部分，是JVM运行时数据区的组成，关注的是数据的存储、分配和回收。JMM是一种规范，定义了线程如何以及何时可以看到其他线程修改过的共享变量的值，关注的是并发编程中变量的访问规则。
    • happens-before 原则是什么，为什么需要 happens-before 原则。
      • happens-before原则是JMM中的一个核心概念，用来保证内存的一致性，无需加锁也能获得一致的访问效果。它定义了一套规则，确定程序中不同操作之间的先后顺序关系。happens-before原则需要因为它为程序员提供了线程安全的内存可见性保证，简化了多线程程序的编写。
  • volatile
    • 什么是 Java 的内存模型，Java 中各个线程是怎么彼此看到对方的变量的？
      • Java内存模型（JMM）定义了线程和主内存之间的抽象关系及规则，确保线程对共享变量的操作在主内存中正确处理，保障可见性、原子性和有序性。线程间通过主内存交换变量，每个线程都有自己的工作内存，变量的更新需要同步回主内存中，其他线程才能看到最新值。
    • 请谈谈 volatile 有什么特点，为什么它能保证变量对所有线程的可见性？
      • volatile 关键字确保变量的修改立即更新到主内存，同时每次使用前直接从主内存刷新，因此能保证变量对所有线程的可见性。它避免了变量在工作内存的线程间不一致的情况。但volatile不保证原子性。
    • 既然 volatile 能够保证线程间的变量可见性，是不是就意味着基于 volatile变量 的运算就是并发安全的？
      • 不是。volatile 可以保证变量的可见性，但不能保证复合操作（如自增运算）的原子性。如果有多个线程同时执行写操作，基于volatile变量的运算仍然可能导致数据竞争和不一致问题。
    • 请对比下 volatile 对比 Synchronized 的异同。
      • 相同点：都能保证可见性，确保更新操作对其他线程是可见的。
      • 不同点：
        • volatile 是轻量级的，只能用于变量级的同步，而 synchronized 可以锁定代码块或对象。
        • volatile 不保证原子性，synchronized 保证了操作的原子性。
        • volatile 不会引起线程上下文切换和调度，但 synchronized 可能会引起线程阻塞。
    • synchronized 和 volatile 的区别讲一下。讲一下 volatile 关键字。
      • synchronized：关键字提供了一种互斥机制，能够保证同一时刻只有一个线程执行特定代码块，保证原子性、可见性和有序性。
      • volatile：关键字保证了变量修改的可见性和部分有序性（禁止指令重排），但不保证原子性。适用于状态标记或发生变化时需要通知其他线程的场景。
      • 区别：volatile 主要用于变量的可见性，而 synchronized 既能保证可见性也能保证原子性。synchronized 相比 volatile 更重量级，使用时需要考虑性能开销。
  • synchronized
    • synchronized 关键字有什么作用，你平时怎么使用 synchronized 的。
      • synchronized 关键字是 Java 中多线程锁的悲观锁的一种应用方式，
      • 一般使用 synchronized 关键字修饰同步方法与同步代码块，此时对该方法/代码块访问的时候会进行加锁，其他线程访问只有等待该线程释放锁后才能进行。
    • 说一下偏向锁、轻量级锁、重量级锁（锁升级过程）。
      • 偏向锁：假设锁总是被同一个线程获取，减少同步开销。当锁未被竞争时启用。
      • 轻量级锁：当偏向锁失败，多个线程交替获取锁时使用。利用CAS操作标记锁状态，避免阻塞。
      • 重量级锁：当多线程同时竞争锁时，升级为重量级锁，涉及到操作系统层面的互斥，有较大开销。
      • 升级过程：偏向锁→轻量级锁→重量级锁，随着竞争增加逐步升级，以适应不同的竞争情况。
    • synchronized 关键字的底层原理讲一下。
      • synchronized 关键字在JVM层面实现同步机制。它依赖于监视器锁（monitor），控制同步块或方法的访问。对象头中的Mark Word记录锁的状态和拥有者。对于同步块，JVM基于进入和退出Monitor对象实现方法锁定和释放
    • 你刚才提到获取对象的锁，这个“锁”到底是什么？如何确定对象的锁？
      • “锁”是一种机制，用于实现线程间的同步，确保同一时刻只有一个线程访问特定资源或代码块。在Java中，每个对象都有一个监视器锁。当线程想要执行同步代码块时，必须先获得该对象的锁。锁的状态存储在对象头的Mark Word中。
    • 什么是可重入性，为什么说 Synchronized 是可重入锁？
      • 可重入性指同一个线程可以多次获取同一把锁。synchronized 是可重入的，因为Java监视器锁的获取计数机制允许同一个线程多次获得同一锁，避免了自锁死的情况。
    • JVM 对 Java 的原生锁做了哪些优化？
      • JVM对synchronized做了多项优化，包括锁偏向、轻量级锁、锁粗化、锁消除、自旋锁等，以减少锁操作的开销，提高多线程程序的执行效率。
    • JDK1.6 之后的 synchronized 关键字底层做了哪些优化。synchronized 锁升级流程。
      • JDK 1.6 引入了锁偏向、轻量级锁和重量级锁的概念，并实现了锁升级机制。锁开始时偏向于第一个获取它的线程，若其他线程尝试获取，升级为轻量级锁；竞争激烈时升级为重量级锁。
    • 为什么说 Synchronized 是非公平锁？
      • synchronized 是非公平锁，因为它不保证等待锁的线程按照请求顺序获取锁。当锁被释放时，任何一个等待锁的线程都有机会获取锁，而不是最早请求的线程。
    • 什么是锁消除和锁粗化？
      • 锁消除：JVM优化掉那些不可能存在共享数据竞争的锁，减少不必要的同步开销。
      • 锁粗化：将多次连接的锁操作合并为一次，扩大锁的范围，减少锁获取和释放的次数
    • 为什么说 Synchronized 是一个悲观锁？乐观锁的实现原理又是什么？什么是 CAS，它有什么特性？
      • synchronized 是悲观锁，因为它假设最坏的情况，每次加锁保护共享资源。乐观锁假设没有冲突，通过版本号等机制在更新时检查是否有冲突。CAS（Compare-And-Swap）是乐观锁的一种实现，它通过比较并交换值来实现非阻塞同步，特性是无锁编程，减少线程阻塞。
    • 乐观锁一定就是好的吗？
      • 乐观锁不总是最佳选择。在高竞争环境下，乐观锁可能导致大量的重试和失败，反而降低性能。选择乐观锁还是悲观锁应根据具体场景、数据访问模式和竞争条件来决定。
  • ReentrantLock
    • 跟 Synchronized 相比，可重入锁 ReentrantLock 其实现原理有什么不同？
      • ReentrantLock 提供了比 synchronized 更灵活的锁定机制。它基于 AQS（AbstractQueuedSynchronizer）框架实现，支持公平锁和非公平锁，提供了条件变量（Condition）支持，允许分离的等待/通知，同时提供了尝试获取锁、可中断获取锁的能力。
    • synchronized 和 ReentrantLock 有什么区别。
      • synchronized 是Java内置关键字，提供自动的锁管理；ReentrantLock 是API层面的，提供更丰富的功能，如可中断等待、公平锁。
      • synchronized 不支持条件变量，ReentrantLock 提供了Condition支持，允许更细粒度的线程同步。
      • ReentrantLock 提供了锁的公平性选择，而 synchronized 总是非公平锁
    • ReentrantLock 是如何实现可重入性的？
      • ReentrantLock 实现可重入性通过记录锁持有线程和持有次数。当线程首次获取锁时，JVM记录锁的持有者，并将持有次数设为1。如果同一线程再次获取锁，持有次数增加。线程释放锁时，次数减少。当次数为0时，锁被释放。
    • 除了 ReetrantLock，你还接触过 JUC 中的哪些并发工具？
      • 除了 ReentrantLock，JUC（java.util.concurrent）包中还包括：
        • CountDownLatch：允许一个或多个线程等待其他线程完成操作。
        • CyclicBarrier：允许一组线程相互等待至某个状态点。
        • Semaphore：控制对共享资源的访问。
        • ConcurrentHashMap：线程安全的HashMap实现。
        • ReadWriteLock：读写锁，允许多个读取者、一个写入者。
    • 请谈谈 ReadWriteLock 和 StampedLock。
      • ReadWriteLock 允许多个线程同时读取，但只有一个线程写入。它通过分离读和写操作来提高性能。
      • StampedLock 是Java 8 引入的，提供了一种乐观读锁的机制，允许获取一个戳(stamp)，然后尝试读取，如果读取期间写锁未被占用，则读取成功，否则可以升级为悲观读锁。它比ReadWriteLock提供了更高的并发性。
  • 线程池
    • 线程池有哪几种，线程池的重要参数有哪些？
      • Java中常见的线程池类型包括：FixedThreadPool、CachedThreadPool、SingleThreadExecutor、ScheduledThreadPoolExecutor。线程池的重要参数有：核心线程数、最大线程数、空闲线程存活时间、任务队列、线程工厂、拒绝策略。
    • Java 中的线程池是如何实现的？创建线程池的几个核心构造参数？
      • Java线程池通过ThreadPoolExecutor类实现，提供灵活的线程池管理策略。其核心构造参数包括：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（非核心线程空闲存活时间）、unit（时间单位）、workQueue（任务队列）、threadFactory（线程工厂）、handler（拒绝策略）。
    • 线程池中的线程是怎么创建的？是一开始就随着线程池的启动创建好的吗？
      • 线程池中的线程不是一开始就创建好的。当任务被提交且核心线程未满时，线程池会创建核心线程来处理任务；如果核心线程已满且任务队列未满，则任务被加入队列等待；如果队列也满了且未达到最大线程数，则创建非核心线程处理任务。线程是按需创建的，以优化资源使用。
    • 既然提到可以通过配置不同参数创建出不同的线程池，那么 Java 中默认实现好的线程池又有哪些呢？请比较它们的异同
      • Java中通过Executors类提供的默认线程池有：newFixedThreadPool（固定大小）、newCachedThreadPool（可缓存）、newSingleThreadExecutor（单线程执行器）、newScheduledThreadPool（定时及周期任务执行）。它们的主要区别在于管理线程的方式和用途，如固定大小线程池用于处理固定数量的任务，而可缓存线程池适合处理大量短期异步任务。
    • 如何在 Java 线程池中提交线程？
      • 在Java线程池中提交任务通常有两种方式：使用execute(Runnable command)方法提交不需要返回值的任务；使用submit(Callable<T> task)或submit(Runnable task, T result)方法提交需要返回值的任务。submit方法返回一个Future<T>对象，可用于检查任务是否完成及获取返回值。
  • AQS
    • AQS 是什么，AQS 的原理是什么？那么请谈谈 AQS 框架是怎么回事儿？
      • AQS（AbstractQueuedSynchronizer）是Java并发包中的一个框架，用于构建锁或其他同步器。AQS使用一个int成员变量表示同步状态，通过内置的FIFO队列来完成资源获取线程的排队工作。同步器的实现依赖于继承AQS并实现其protected方法来管理同步状态
    • 什么是ABA问题，怎么解决？
      • ABA问题是在CAS操作中出现的，指一个位置的值原先是A，变成了B，然后又变回A，CAS操作无法感知中间的变化。解决ABA问题的一种方法是使用“版本号”，如Java的AtomicStampedReference，通过维护一种带有时间戳/版本号的标记来判断值是否真正没有变过。
    • CAS 了解吗，原理说一下。
      • CAS（Compare-And-Swap）是一种无锁算法，它包括三个参数：内存位置、预期原值和新值。只有当内存位置的值与预期原值相匹配时，才会将内存位置的值更新为新值，这个操作是原子的。CAS是实现同步原语（如锁和原子变量）的基础。
    • 如何让 Java 的线程彼此同步？你了解过哪些同步器？请分别介绍下。
      • Java线程同步可以通过synchronized关键字、Lock接口实现的锁（如ReentrantLock）、以及CountDownLatch、CyclicBarrier、Semaphore等并发工具实现。synchronized提供隐式锁定，而Lock提供了更复杂的锁操作。CountDownLatch允许一个或多个线程等待其他线程完成操作；CyclicBarrier允许一组线程相互等待到达一个公共屏障点；Semaphore管理一组许可证以控制对资源的访问
    • Semaphore 有什么用，他的原理是什么？
      • Semaphore是一种基于计数的同步工具，可以控制同时访问某个特定资源的操作数量。它的原理是维护一组许可证，线程在执行操作前尝试从Semaphore获取许可证，如果许可证数量大于0，则线程可以获取许可证并执行；如果等于0，则线程阻塞直到有许可证被释放。
    • CountDownLatch，CyclicBarrier 了解吗。CyclicBarrier 和 CountDownLatch 看起来很相似，请对比下呢？
      • CountDownLatch允许一个或多个线程等待其他线程完成操作。它不能重置计数器，一旦计数结束，所有等待的线程被释放。
      • CyclicBarrier允许一组线程相互等待至某个公共屏障点。CyclicBarrier可以重用，当所有等待线程到达屏障时，屏障会自动重置。
      • 对比：CountDownLatch是一次性的，CyclicBarrier可以重用。CountDownLatch主要用于事件完成时触发动作，而CyclicBarrier用于多线程间相互等待。
  • ThreadLocal
    • ThreadLocal 的底层原理讲一下，你在平时的项目中是如何使用 ThreadLocal 关键字的。
      • ThreadLocal提供线程局部变量，每个线程都有该变量的独立副本。底层通过在Thread对象中维护一个ThreadLocal.ThreadLocalMap，Map的键是ThreadLocal对象，值是线程的变量副本。在项目中，经常用ThreadLocal来保存数据库连接、会话信息等线程相关的数据，确保数据隔离且线程安全。
    • 很多人都说要慎用 ThreadLocal，谈谈你的理解，使用 ThreadLocal 需要注意些什么。
      • ThreadLocal应慎用，因为它使用不当容易导致内存泄漏。ThreadLocal变量在不再使用时应及时调用remove()方法清理，避免长生命周期的线程（如线程池中的线程）持有ThreadLocal变量后，导致无法及时回收。同时，应避免在高负载下创建大量ThreadLocal变量，以减少对系统性能的影响。
    • 请谈谈 ThreadLocal 是怎么解决并发安全的？
      • ThreadLocal通过为每个线程提供变量的独立副本解决并发安全问题。由于每个线程访问自己内部ThreadLocalMap中的变量，完全隔离于其他线程，因此不会存在多线程间变量共享的问题，避免了并发访问冲突，从而实现了并发安全。这种方式不需要额外的同步措施，减少了锁竞争，提高了并发性能。
