消息队列_基础概念

大纲

  • 消息队列的核心使用场景
  • 技术选型
  • JMS和AMQP
  • RPC 和消息队列的区别

Start

在面试的时候，当你的简历上面写了用过MQ ,那么面试官的面试这个时候便开始了：

    1. 你在系统里用过消息队列吗
    1. 你们在项目里是如何使用消息队列的。
    1. 为什么你们这里要使用消息队列呢，为什么不直接调用接口处理呢。
    1. 说说消息队列有哪些优点和缺点吧。
    1. 消息队列的选型，他们之间的区别。
    1. 如何保证消息不被重复消息，如何保证消费的时候是幂等，如何保证消息的可靠性，如何保证消息的顺序性。
    1. 如何解决消息队列的延时以及过期失效问题？消息队列满了以后该如何处理？有几百万消息持续堆积几小时，要怎么解决？
    1. 让你去设计一个MQ，该如何进行架构设计，说一下你的思路。

前几个问题的思考主要是围绕：为什么要使用消息队列；

陈述回答的时候需要从消息队列利弊、优缺点、风险点等方面考虑，在进行技术选型的时候，主要还是围绕使用场景来决定。

每一个 MQ 没有绝对的好坏，但是要看用在哪个场景可以扬长避短，利用其优势，规避其劣势。

1、消息队列的核心使用场景

消息队列（MQ）的核心使用场景主要包括三个方面：解耦、异步处理和削峰。

解耦

类比解释

想象一下，你在组织一个大型派对，你需要通知很多朋友。如果你直接一个个打电话，每当新朋友加入或某人退出，你都需要更新你的通知列表。这很麻烦，对吧？

现在，假设你创建了一个在线活动页面，你只需要更新这个页面，而你的朋友们自己决定是否查看这个页面。这就大大简化了你的工作。

技术解释

在没有消息队列的情况下，系统 A 需要直接和多个系统（如 B、C、D、E）进行通信。这种紧密耦合的方式使得系统 A 需要不断更新与其他系统的交互逻辑。使用消息队列后，系统 A 只需将消息发布到队列，而其他系统根据需要从队列中取出消息。

这种发布/订阅模式（Pub/Sub）使得系统 A 与其他系统解耦，简化了维护和扩展的复杂性。

异步处理

类比解释

假设你在一家餐厅点餐。如果厨师需要等待一个菜做完才开始做下一个，这将非常低效。但如果他们能同时处理多个订单，每道菜完成时就直接上菜，这样效率会更高。

技术解释

在同步处理模式中，系统 A 在完成所有操作（如更新多个数据库）之前不会响应用户请求，导致延迟。
而异步处理使用消息队列，系统 A 可以快速响应用户请求，将后续任务（如数据库更新）放入队列中，由其他系统或服务慢慢处理。这减少了用户的等待时间，提高了系统的响应能力。

削峰

类比解释

想象一下，一个公交车站在高峰时段面临大量乘客。如果每个人都立即上车，公交车会被迅速塞满。但如果有一个等候区，让乘客按顺序上车，这样既保持了公交车的正常运行，又避免了拥挤。

技术解释

在流量削峰场景中，消息队列充当缓冲区，帮助处理短时的高流量。当系统 A 在高峰时段面临大量请求时，这些请求首先被放入消息队列。
系统 A 根据自己的处理能力逐渐从队列中取出并处理这些请求。这样可以避免在高峰期对系统造成过大的压力，同时保证了系统的稳定性和可用性。

上面提及到的削峰、解耦、异步就是在特殊场景下其对应的好处，而缺点对应的是系统的复杂性变高，可用性降低，以及还有一致性问题。

使用消息队列会带来哪些问题？

  • 系统可用性降低： 系统可用性在某种程度上降低，为什么这样说呢？在加入 MQ 之前，你不用考虑消息丢失或者说 MQ 挂掉等等的情况，但是，引入 MQ 之后你就需要去考虑了！
  • 系统复杂性提高： 加入 MQ 之后，需要保证消息没有被重复消费、处理消息丢失的情况、保证消息传递的顺序性等等问题！
  • 一致性问题： 消息队列可以实现异步，消息队列带来的异步确实可以提高系统响应速度。但是，万一消息的真正消费者并没有正确消费消息怎么办？这样就会导致数据不一致的情况

Kafka

优点

  1. 高吞吐量：Kafka 设计用于处理高吞吐量数据，适合大规模消息处理应用。
  2. 分布式系统：它天生就是分布式的，易于扩展。
  3. 持久性和可靠性：Kafka 可以持久化消息到磁盘，因此对消息不会因为系统故障而丢失。
  4. 高效：Kafka 使用顺序磁盘I/O，提供高性能与数据吞吐。

缺点

  1. 复杂性：Kafka 的设置和管理相对复杂。
  2. 消息积压：在高负载下，消息延迟可能会增加。
  3. 不支持优先级队列。

ActiveMQ

优点

  1. 支持多种传输协议：如 OpenWire, STOMP, MQTT 等。
  2. 易于使用和配置：相比 Kafka，ActiveMQ 配置和管理较为简单。
  3. JMS 支持：完全支持 JMS API 和 J2EE 集成。

缺点

  1. 较低的吞吐量：相比 Kafka，ActiveMQ 的吞吐量较低。
  2. 可扩展性问题：在非常大的数据负载下，扩展性可能是一个问题。

RabbitMQ

优点

  1. 灵活的路由：支持复杂的路由。
  2. 多种消息协议：支持 AMQP, MQTT, STOMP 等。
  3. 管理界面友好：提供易于使用的管理界面。

缺点

  1. 性能问题：在高吞吐量时，性能可能会受限。
  2. 持久化开销：消息持久化可能会导致性能问题。

RocketMQ

优点

  1. 高性能和低延迟：在分布式系统中表现出色。
  2. 丰富的功能：提供了大量的特性，如消息顺序、事务消息等。
  3. 可靠性和扩展性：保证了消息的可靠传输，同时支持水平和垂直扩展。

缺点

  1. 社区和生态：相比 Kafka，RocketMQ 的社区和生态相对较小。
  2. 复杂性：配置和管理稍微复杂。

技术选型考虑因素

  1. 吞吐量需求：如果你的系统需要处理大量的数据，Kafka 或 RocketMQ 可能是更好的选择。
  2. 系统复杂性：如果你需要一个易于设置和管理的系统，ActiveMQ 或 RabbitMQ 可能更合适。
  3. 功能需求：根据你需要的特定功能（如消息排序、事务处理等）选择相应的系统。
  4. 社区支持和生态系统：强大的社区可以提供更好的支持和更多的资源。
  5. 可靠性和持久性：如果这些是你的主要关注点，Kafka 和 RocketMQ 是不错的选择

选择建议：

  • 中小型公司，技术实力较为一般，技术挑战不是特别高，用 RabbitMQ 是不错的选择；
  • 大型公司，基础架构研发实力较强，用 RocketMQ 是很好的选择。
  • 如果是大数据领域的实时计算、日志采集等场景，Kafka 是业内标准，社区活跃度很高，几乎是全世界该领域的事实性规范。
  • ActiveMQ，最近逐年使用减少，没经过大规模吞吐量场景的验证，社区不是很活跃。

3、JMS和AMQP

JMS（Java Message Service）和 AMQP（Advanced Message Queuing Protocol）都是消息传递的标准和规范，但它们在设计理念、实现方式和使用场景上有所不同

JMS（Java Message Service）

定义

  • JMS 是一个 Java 平台中关于面向消息中间件（MOM）的 API，用于在两个应用程序之间，或分布式系统中发送消息，进行异步通信。

特点

  • Java 标准：JMS 是 Java EE 的一部分，主要用于 Java 应用程序。
  • API 规范：定义了一个通用的 API 来与消息服务进行交互，而不是一个网络协议。
  • 提供者依赖：JMS API 可以由多种供应商实现，如 ActiveMQ、IBM MQ 等。
  • 两种消息模型：支持点对点（Queue）和发布/订阅（Topic）模型。
  • 不跨语言：主要用于 Java 应用程序，不适用于不同语言之间的通信。

AMQP（Advanced Message Queuing Protocol）

定义

  • AMQP 是一个开放标准的应用层协议，专为异步消息传递提供一个统一的框架，支持跨平台和跨语言的消息通信。

特点

  • 协议标准：AMQP 定义了网络协议，包括消息的格式、传输方式和队列的行为。
  • 供应商中立：旨在允许从不同供应商的消息系统中互操作。
  • 跨语言支持：不限于 Java，支持多种编程语言和平台。
  • 多种消息模式：支持灵活的消息模式，包括点对点、发布/订阅、请求/响应等。
  • 可靠性和安全性：提供了消息确认、交易、安全等机制。

主要区别

  • 设计范围：JMS 是一个针对 Java 的 API 规范，而 AMQP 是一个跨语言、跨平台的网络协议。
  • 适用范围：JMS 主要用于 Java 应用，AMQP 适用于任何支持该协议的客户端。
  • 实现依赖：使用 JMS 时，需要依赖特定供应商的实现，而 AMQP 的实现是供应商中立的，理论上任何 AMQP 兼容的客户端都可以与任何 AMQP 兼容的服务器通信。
  • 兼容性：AMQP 的设计目标是实现不同消息系统之间的互操作性，而 JMS 无法做到这一点。

ActiveMQ（已被淘汰） 是基于 JMS 规范实现的。

RabbitMQ 是基于 AMQP 协议实现的。

4、RPC 和消息队列的区别

RPC（Remote Procedure Call）和消息队列是分布式系统中的两种不同的通信机制，它们在设计目的、通信方式、架构要求和时效性等方面存在显著差异。

用途

  • RPC：
    • 目的是实现不同服务之间的直接同步调用。
    • 类似于本地函数调用，用于服务与服务之间的紧密协作。
    • 适用于对响应时间有严格要求的场景。
  • 消息队列：
    • 目的是提供异步通信能力，解耦服务之间的依赖关系。
    • 用于任务的异步处理、流量削峰和分布式事务等。
    • 适用于处理高峰期流量或者执行耗时操作。

通信方式

  • RPC：
    • 是一种同步操作，调用方发起请求并等待响应，直到远程过程执行完成并返回结果。
    • 依赖直接的网络连接和即时的数据传输。
  • 消息队列：
    • 提供了一种异步通信方式，发送方将消息发送到队列，接收方可以在任何时间从队列中获取消息。
    • 引入了中间件作为消息传递的中间层，增加了消息的持久性。

架构

  • RPC：
    • 没有消息存储的需求，因为它依赖即时的请求-响应模型。
    • 通常不需要中间层来缓存或管理消息。
  • 消息队列：
    • 必须要有消息存储和管理的机制，因为消息需要在队列中等待被处理。
    • 消息中间件负责保证消息的持久性和可靠性。

时效性

  • RPC：
    • 一般期望立即处理，调用方通常会阻塞等待直到得到响应。
    • 更适合于需要快速完成的交互式操作。
  • 消息队列：
    • 消息可能不会立即处理，允许在系统资源允许的情况下随后处理。
    • 支持高峰期的流量平滑，允许系统按照自己的节奏和能力处理消息。

总结

RPC 和消息队列解决的是分布式系统中不同的问题。

RPC 强调的是即时性和同步，适合于需要迅速响应的场景。而消息队列强调的是异步性和解耦，适合于需要提高系统整体效率和稳定性的场景。

在实际应用中，两者往往相互补充，共同支撑起一个高效、可靠、可扩展的分布式系统架构。
